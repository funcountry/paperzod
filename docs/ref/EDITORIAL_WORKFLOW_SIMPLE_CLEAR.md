# Editorial Workflow Simple Clear

## Purpose

This is the smallest plain-English statement of the editorial workflow the
canonical public example wants to model.

## Workflow

1. `Brief Researcher`
   - define what the piece needs to communicate
   - gather the evidence and source notes the next lane may trust
   - produce `EDITORIAL_BRIEF.md`

2. `Story Architect`
   - turn the brief into a concrete story outline
   - preserve upstream constraints
   - produce `STORY_OUTLINE.md`

3. `Editorial Acceptance Gate`
   - verify the outline is trustworthy
   - verify the shared quality bar still holds
   - accept or send back with precise feedback

## Shared Doctrine We Need

The runtime tree should have:

- role homes
- a shared entrypoint
- one authoritative workflow owner
- lane workflow docs
- shared standards
- a gate doc
- supporting references

## Why This Doc Exists

This file is imported as grounding material by the public editorial setup.

It is intentionally short. Its job is to explain the workflow shape clearly,
not to become the runtime doctrine itself.
