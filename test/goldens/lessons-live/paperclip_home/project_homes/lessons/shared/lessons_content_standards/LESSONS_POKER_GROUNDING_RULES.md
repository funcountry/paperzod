# Lessons Poker Grounding Rules

This file owns one question: how does Lessons ground poker meaning and
poker-native wording before a concept claim, wording choice, or learner-facing
line is trusted?

Use `technical_references/POKER_KB.md` for the runner path and query mechanics.
Use `lessons_content_standards/LESSON_RIGHT_MOVE_RULES.md` if the lesson tells the
learner the right move in a specific hand.

## Core Rule

Do not fake poker grounding.

If a concept, wording choice, or learner-facing line needs poker grounding,
leave real grounding or stop and say the grounding is missing.

## What Books Are For

Use Books for:

- correctness
- definitions
- formal framing
- the semantic skeleton of the claim

## What Forums Are For

Use Forums for:

- player phrasing
- coaching wording
- lexical sanity checks
- checking whether the line sounds like a real player or coach

## What Counts As Enough Grounding

- For concept and term decisions:
  - leave enough Books grounding to defend the meaning
  - add Forums when learner-facing wording depends on real player language
- For learner copy:
  - use Books for the semantic skeleton
  - use Forums for the player-language pass
- For hints, feedback, and short explanations:
  - leave enough grounding to prove the deciding variable
  - keep the wording in real poker language
- If one source is honestly not needed, say why in the packet instead of
  silently skipping it

## What The Packet Should Name

Name all of these when grounding matters:

- the claim or line you were grounding
- the source or sources you used
- the successful query
- the wording, concept, or framing decision that came out of the grounding

## Stop And Escalate Rule

If PokerKB cannot return usable grounding:

- say exactly what query failed
- include the failed output when available
- block the claim or narrow the scope instead of smoothing over the gap
